# Database Migrations Guide

## 📊 Sistema de Tracking de Migraciones

Prisma mantiene un **historial completo** de todas las migraciones de forma automática:

### 1. Archivos de Migración (versionados en Git)
**Ubicación:** `packages/database/prisma/migrations/`

Cada migración genera:
```
migrations/
├── migration_lock.toml          # Lock del provider (PostgreSQL)
├── 20260130022053_init/         # Timestamp + nombre descriptivo
│   └── migration.sql            # SQL ejecutado
├── 20260130150000_add_roles/    # Nueva migración
│   └── migration.sql
└── ...
```

### 2. Tabla de Control (`_prisma_migrations`)
Prisma crea automáticamente esta tabla en tu base de datos para trackear:

| Campo | Descripción |
|-------|-------------|
| `id` | ID único de la migración |
| `checksum` | Hash del archivo SQL para detectar cambios |
| `finished_at` | Timestamp de ejecución |
| `migration_name` | Nombre de la carpeta de migración |
| `logs` | Logs de ejecución |
| `rolled_back_at` | Timestamp si fue revertida |
| `started_at` | Inicio de ejecución |
| `applied_steps_count` | Número de pasos aplicados |

## 🔧 Comandos Disponibles

### Desarrollo

#### Crear Nueva Migración (automático)
```bash
# 1. Edita el schema.prisma
# 2. Ejecuta:
pnpm --filter database db:migrate --name nombre_descriptivo

# Prisma:
# - Detecta cambios en el schema
# - Genera el SQL necesario
# - Crea la carpeta con timestamp
# - Aplica la migración
# - Actualiza Prisma Client
```

#### Crear Migración Sin Aplicar
```bash
pnpm --filter database db:migrate:create --name nombre_descriptivo

# Útil para:
# - Revisar el SQL antes de aplicar
# - Modificar manualmente la migración
# - Código review
```

#### Ver Estado de Migraciones
```bash
pnpm --filter database db:migrate:status

# Muestra:
# - Migraciones aplicadas ✓
# - Migraciones pendientes
# - Estado del schema
# - Drift (cambios manuales en DB)
```

#### Ver Historial
```bash
pnpm --filter database db:migrate:history

# Igual que status, muestra el historial completo
```

#### Reset Completo (⚠️ Peligroso)
```bash
pnpm --filter database db:reset

# Elimina TODO:
# - Borra la base de datos
# - Recrea desde cero
# - Aplica todas las migraciones
# - Ejecuta seeds
# 
# ⚠️ Solo en desarrollo!
```

### Producción

#### Aplicar Migraciones Pendientes
```bash
pnpm --filter database db:migrate:deploy

# Aplica solo las migraciones nuevas
# No genera nuevas migraciones
# No modifica el schema
# Usado en CI/CD
```

## 🎯 Flujo de Trabajo Típico

### Escenario 1: Agregar un Campo

```bash
# 1. Editar schema.prisma
# packages/database/prisma/schema.prisma
model User {
  id        Int     @id @default(autoincrement())
  email     String  @unique
  name      String?
  role      String  @default("user")  // ← NUEVO CAMPO
  createdAt DateTime @default(now())
}

# 2. Crear y aplicar migración
pnpm --filter database db:migrate --name add_user_role

# Output:
# ✔ Migration created: 20260130123456_add_user_role
# ✔ Applied migration: 20260130123456_add_user_role
# ✔ Generated Prisma Client

# 3. Ver el SQL generado
# cat packages/database/prisma/migrations/20260130123456_add_user_role/migration.sql
# ALTER TABLE "User" ADD COLUMN "role" TEXT NOT NULL DEFAULT 'user';

# 4. Commit a Git
git add packages/database/prisma/migrations/
git add packages/database/prisma/schema.prisma
git commit -m "feat: add user role field"
```

### Escenario 2: Crear Nueva Tabla

```bash
# 1. Agregar model al schema
model Product {
  id          Int      @id @default(autoincrement())
  name        String
  price       Decimal  @db.Decimal(10, 2)
  userId      Int
  user        User     @relation(fields: [userId], references: [id])
  createdAt   DateTime @default(now())
}

model User {
  // ... campos existentes
  products    Product[]  // ← Relación
}

# 2. Crear migración
pnpm --filter database db:migrate --name create_products_table

# 3. Verificar que se aplicó
pnpm --filter database db:migrate:status
# Output: Database schema is up to date!
```

### Escenario 3: Modificación Compleja (revisar SQL)

```bash
# 1. Crear migración sin aplicar
pnpm --filter database db:migrate:create --name complex_data_migration

# 2. Editar manualmente el SQL generado
# packages/database/prisma/migrations/[timestamp]_complex_data_migration/migration.sql

-- Generated by Prisma
ALTER TABLE "User" ADD COLUMN "status" TEXT;

-- Custom: Migrar datos existentes
UPDATE "User" SET "status" = 'active' WHERE "isActive" = true;
UPDATE "User" SET "status" = 'inactive' WHERE "isActive" = false;

-- Hacer campo obligatorio después de migrar
ALTER TABLE "User" ALTER COLUMN "status" SET NOT NULL;
ALTER TABLE "User" DROP COLUMN "isActive";

# 3. Aplicar la migración editada
pnpm --filter database db:migrate --name complex_data_migration
```

### Escenario 4: Resolver Conflictos

```bash
# Si alguien más creó una migración y hay conflicto:

# 1. Pull del repo
git pull origin main

# 2. Ver estado
pnpm --filter database db:migrate:status

# Output: Database contains migrations not found locally

# 3. Aplicar migraciones nuevas
pnpm --filter database db:migrate

# 4. Si hay conflictos en el schema, resolverlos y crear nueva migración
```

## 🚀 Deployment en Producción

### Pipeline de CI/CD Típico

```yaml
# .github/workflows/deploy.yml
steps:
  - name: Install dependencies
    run: pnpm install --prod

  - name: Generate Prisma Client
    run: pnpm --filter database db:generate:prod
    env:
      DATABASE_URL: ${{ secrets.DATABASE_URL }}

  - name: Run migrations
    run: pnpm --filter database db:migrate:deploy
    env:
      DATABASE_URL: ${{ secrets.DATABASE_URL }}

  - name: Build application
    run: pnpm run build

  - name: Deploy
    run: # ... deploy steps
```

## 🔍 Comandos de Inspección

### Ver SQL de una Migración
```bash
cat packages/database/prisma/migrations/[timestamp]_[name]/migration.sql
```

### Ver Todas las Migraciones en la DB
```sql
-- Conectarse a la DB y ejecutar:
SELECT * FROM "_prisma_migrations" ORDER BY "started_at" DESC;
```

### Verificar Drift (cambios manuales)
```bash
pnpm --filter database db:migrate:status

# Si hay drift:
# Warning: Your database schema is not in sync with your Prisma schema
```

## 📋 Best Practices

### ✅ DO

1. **Nombres descriptivos** para las migraciones
   ```bash
   pnpm db:migrate --name add_user_avatar_field
   # NO: pnpm db:migrate --name update
   ```

2. **Commits atómicos** - una migración por commit
   ```bash
   git add prisma/migrations/[timestamp]_add_feature/
   git add prisma/schema.prisma
   git commit -m "feat: add feature to schema"
   ```

3. **Review del SQL** generado antes de aplicar en producción
   ```bash
   pnpm db:migrate:create --name change
   # Revisar el SQL
   # Aplicar manualmente
   ```

4. **Backup** antes de migraciones grandes en producción
   ```bash
   pg_dump database > backup_$(date +%Y%m%d).sql
   ```

5. **Testear migraciones** en staging primero
   ```bash
   # En staging
   pnpm db:migrate:deploy
   # Verificar que todo funciona
   # Luego aplicar en producción
   ```

### ❌ DON'T

1. **NO editar migraciones ya aplicadas**
   - Crear nueva migración en su lugar
   - Las ya aplicadas son "immutable"

2. **NO hacer `db:push` en producción**
   - Solo para prototipado rápido
   - Usar `db:migrate` siempre

3. **NO modificar la DB manualmente**
   - Todo cambio debe pasar por Prisma Migrate
   - Evita drift entre schema y DB

4. **NO usar `db:reset` en producción**
   - Borra TODOS los datos
   - Solo para desarrollo

5. **NO commitear `.env`**
   - Ya está en `.gitignore`
   - Solo commitear `.env.example`

## 🐛 Troubleshooting

### "Migration already applied"
```bash
# La migración ya existe en DB pero no localmente
# Solución: pull del repo
git pull origin main
pnpm --filter database db:migrate:status
```

### "Database schema drift detected"
```bash
# La DB tiene cambios manuales
# Opción 1: Resetear DB (desarrollo)
pnpm --filter database db:reset

# Opción 2: Crear migración baselining (producción)
pnpm --filter database db:migrate:resolve --applied [migration_name]
```

### "Migration failed"
```bash
# Ver logs en _prisma_migrations
SELECT * FROM "_prisma_migrations" WHERE "finished_at" IS NULL;

# Marcar como aplicada si es necesario
pnpm --filter database db:migrate:resolve --applied [migration_name]

# O marcar como fallida para reintentar
pnpm --filter database db:migrate:resolve --rolled-back [migration_name]
```

## 🎓 Ejemplo Completo: Feature Branch

```bash
# 1. Crear branch
git checkout -b feature/add-products

# 2. Modificar schema
# Editar packages/database/prisma/schema.prisma

# 3. Crear migración
pnpm --filter database db:migrate --name add_products_table

# 4. Testear cambios
pnpm run dev
# Verificar que todo funciona

# 5. Commit
git add packages/database/prisma/
git commit -m "feat: add products table with relations"

# 6. Push
git push origin feature/add-products

# 7. Merge a main (después del review)
git checkout main
git merge feature/add-products

# 8. En producción (automático en CI/CD)
pnpm --filter database db:generate:prod
pnpm --filter database db:migrate:deploy
```

Tu sistema de migraciones está completamente trackeado y versionado! 🎉
